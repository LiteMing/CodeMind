## Findings

### 1. 高：AI 拓展如果只新增关系线，会被静默丢弃

- 位置：
  - `internal/server/server.go:2175`
  - `internal/server/server.go:2229`
  - `internal/server/server.go:2255`
- 问题：
  - `mergeGeneratedGraphIntoDocument` 会先把新关系追加到 `doc.Relations`。
  - 但只要 `addedNodes == 0`，函数就直接 `return base, nil`。
  - 这会把刚追加的新关系全部丢掉。
- 影响：
  - 当模型返回“只补充 cross-link，不新增节点”时，后端会返回原文档，用户看起来像是“AI 成功执行但没有结果”。
  - 如果新增节点因为后面的标题去重逻辑被吞掉，这个分支也会一起吞掉原本合法的新关系。
- 现状：
  - 现有测试覆盖了扩展新增节点的场景，但没有覆盖“仅新增关系”的场景。

### 2. 中：AI 拓展按标题做全局去重，会把合法重名节点错误合并

- 位置：
  - `internal/server/server.go:2181`
  - `internal/server/server.go:2293`
  - `internal/server/server.go:2345`
- 问题：
  - `titleToID` 以“标题小写”作为全局键建立映射。
  - `appendExpandedNode` 发现同名标题后，直接把新节点映射到旧节点并跳过新增。
  - `resolveExpansionNodeReference` 还会继续用标题兜底解析关系目标。
- 影响：
  - 当前文档模型并不禁止重名节点，`Document.Validate()` 也没有此限制。
  - 这意味着像“成本 / 风险 / 示例 / 应用”这类常见标题，在不同分支下是合法且常见的。
  - 一旦 AI 拓展返回同名节点，结果可能是：
    - 新节点被直接吞掉；
    - 关系线错误地连到另一个同名节点；
    - 拓展结果落到错误分支，但接口仍返回 200。
- 现状：
  - 现有测试没有覆盖“文档中存在同名节点”的扩展场景。

### 3. 中：保存接口返回的是旧请求体，不是持久化后的文档

- 位置：
  - `internal/server/server.go:313`
  - `internal/store/file_store.go:238`
  - `frontend/src/app.ts:3865`
- 问题：
  - `PUT /api/maps/:id` 中，后端调用 `store.Save(doc)` 后直接把原始 `doc` 返回给前端。
  - 但 `saveLocked` 会在内部调用 `PrepareForSave(time.Now().UTC())`，实际落盘的 `meta.lastEditedAt` 已经更新。
  - 前端 `persistExpandedDocument` 又会立刻用这个响应结果 `openLoadedDocument`，因此内存中的当前文档元数据仍然是旧值。
- 影响：
  - 当前界面主要依赖列表摘要，暂时没有直接炸出用户可见错误。
  - 但只要后续功能开始依赖当前文档的 `meta.lastEditedAt / lastOpenedAt`，就会出现“已保存但内存状态还是旧时间”的不一致。
  - 这类问题很容易在后续增量功能里变成隐蔽 bug。
- 现状：
  - 测试只验证了文件已正确落盘，没有验证 `PUT` 响应体是否同步返回持久化后的最新元数据。
