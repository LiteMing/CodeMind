## Findings

### 1. 高：资源补丁脚本把未渲染的 manifest 模板重新写进了 EXE

- 位置：
  - `scripts/package-desktop.ps1:34`
  - `scripts/patch_windows_resources/main.go:146`
  - `build/windows/wails.exe.manifest:3`
- 问题：
  - 打包脚本会在 `wails build -nopackage` 之后再次调用 `patch_windows_resources`。
  - `applyManifest` 直接读取 `build/windows/wails.exe.manifest`，而这个文件仍然保留了 `{{.Name}}`、`{{.Info.ProductVersion}}` 模板占位符。
  - 结果不是“保留 Wails 已渲染好的 manifest”，而是把模板文件重新解析后写回 EXE。
- 已验证现象：
  - 对现有产物 `build/bin/CodeMind-1.0.0.exe` 执行二进制检索，命中了：
  - `<assemblyIdentity type="win32" name="com.wails.{{.Name}}" version="0.0.0.0" processorArchitecture="*"/>`
- 影响：
  - 当前发布产物已经携带错误的 assembly identity。
  - 这会让桌面包的 manifest 元数据与预期不一致，后续涉及 Windows 识别、资源排查、manifest 相关兼容性时都会变得不可靠。
- 现状：
  - 现有检查只验证了图标和 `ProductVersion`，没有验证 manifest 内容本身。

### 2. 中：版本号存在双份来源，下一次升级版本会直接打包失败

- 位置：
  - `wails.json:12`
  - `scripts/package-desktop.ps1:8`
  - `scripts/package-desktop.ps1:59`
  - `build/windows/info.json:3`
  - `build/windows/info.json:8`
- 问题：
  - 打包脚本把 `wails.json.info.productVersion` 当作目标版本和最终校验值。
  - 但写入 EXE 版本资源的数据来自 `build/windows/info.json`，其中版本号被硬编码为 `1.0.0 / 1.0.0.0`。
  - 只要后续按发布流程把 `wails.json` 改成 `1.0.1`、`1.1.0` 或 `2.0.0`，这里就会产生不一致。
- 影响：
  - 下一次正常版本升级时，`npm run build:desktop` 会在最终版本校验处失败。
  - 发布流程事实上变成了“每次发版必须手工同步两处版本号”，与文档里要求的版本递增流程不匹配。
- 现状：
  - 当前版本恰好仍是 `1.0.0`，所以这类问题不会在日常构建里暴露出来。

### 3. 中：新增节点位置计算忽略 badge 导致的真实高度，容易出现垂直重叠

- 位置：
  - `frontend/src/document.ts:154`
  - `frontend/src/document.ts:171`
  - `frontend/src/document.ts:191`
  - `frontend/src/document.ts:252`
  - `frontend/src/node-sizing.ts:40`
  - `frontend/src/node-sizing.ts:71`
  - `frontend/src/app.ts:2550`
- 问题：
  - `nextStackedY` 在计算下一个兄弟节点 / 子节点 / 浮动节点的纵向位置时，固定调用 `estimateNodeHeight(node, 0)`。
  - 但新的节点高度估算逻辑已经把 `priority` 和子节点数 badge 计入宽度与换行计算。
  - 也就是说，真实渲染高度和这里用于排布的高度已经不再一致。
- 影响：
  - 一旦已有节点带优先级标签或子节点 badge，后续通过新增兄弟节点、子节点或浮动节点得到的位置可能偏高，出现垂直挤压甚至重叠。
  - 用户通常要靠拖拽或手动整理布局才能恢复正常，属于可见回归。
- 现状：
  - 当前仓库没有前端自动化测试覆盖这些布局分支，现有构建也不会发现这种问题。

3. 最后修复 Finding 3，并补一个覆盖 badge 场景的前端布局回归测试或最小复现验证脚本。
